paprLCD

A single source of truth for every pixel and every token.

This document is the canonical reference for all visual decisions in paprLCD. Every color, typeface, size, spacing value, and component is defined here with three names: a CSS variable, a semantic design token, and a SwiftUI code name. If a value you need isn't in this document, you must propose it before using it.

Design principles
01 — One Surface

The same background color everywhere. No surface lifting, no card fills that differ from the page. Depth is expressed through borders and contrast alone.

02 — Monochrome

The palette is a single sage green-grey across all surfaces and ink. No accent colors, no tonal fills. The eye rests on content, not the UI.

03 — Orange for Status Only

Orange is the only color deviation. It is used exclusively for attention, notifications, and urgent states. It is never used for interactive affordances.

04 — Flat Elevation

Zero elevation. No shadows, no blur, no glow. Cards are ink-stroke containers only — a border on the background.

05 — Inversion for Interaction

Interactive elements use a dark background with light text — the "pressed LCD segment" metaphor. Hover stays light with --ink-10.

Governance

Governance

The design system is not a suggestion. Every value in the codebase must be traceable to a named token. This page is the only approved source of truth.

The Rule

Every color, size, spacing value, font, border radius, and animation in the paprLCD codebase must reference a named token from this document. Hard-coded values (#hex, px literals, font strings) are not permitted. If the token you need does not exist here, stop and go through the proposal process before writing the code.

Using tokens in code
CSS / Web
/* ✓ Correct */
background: var(--paper);
color: var(--ink-60);
border: 1px solid var(--border);
font-family: var(--font-mono);
border-radius: var(--radius-md);
transition: background var(--duration-fast) var(--ease-out);

/* ✗ Not allowed */
background: #E8EDE5;
color: rgba(28,28,26,0.68);
border-radius: 8px;
SwiftUI
// ✓ Correct
.background(Color.hb.surfaceBase)
.foregroundColor(Color.hb.textSecondary)
.border(Color.hb.borderDefault)
.font(Font.hb.mono)
.cornerRadius(Radius.hb.md)
.animation(Animation.hb.easeOut)

// ✗ Not allowed
.background(Color(hex: "#E8EDE5"))
.foregroundColor(.gray.opacity(0.6))
.cornerRadius(8)
Approval matrix
No approval needed
  • Using any existing named token in a new context
  • Composing existing components into new layouts
  • Writing or updating documentation
  • Adjusting padding/gap using spacing tokens
  • Choosing between existing button or card variants
  • Using existing icon paths in new places
Requires approval
  • Any new color — even a single new hex value or opacity
  • Any spacing value not in the --space-* scale
  • Any new component or component variant
  • Any animation that doesn't use existing easing + duration tokens
  • Adding a new font weight or size outside the type scale
  • Any new icon (must be added to the icon registry here)
  • Changing a token's value (even minor tweaks)
  • Using orange for anything interactive (buttons, links, nav)
Component & token proposal process

Before adding anything new to the system, work through these five steps. The goal is to discover that existing tokens already cover the need before introducing new ones.

1
Describe the need
Write one sentence: what interaction or visual pattern requires a value that doesn't exist in this document? Be specific — "I need a softer border" is not a description; "The room cell hover state needs a 1px dashed border that reads lighter than --border" is.
2
Check existing token coverage
Search the Token Reference table. Can the need be solved by an existing token used differently, or by combining two existing tokens? If yes — use them. Only continue if the gap is real.
3
Draft the spec
Name the token (follow existing naming conventions), define its value in both LCD light and LCD dark, list which existing tokens it builds on, describe all states (if a component), and write the usage rule ("use only when…").
4
Get design-owner approval
Share the spec in the project backlog. The design owner reviews whether the new token is necessary, whether it fits the existing system hierarchy, and whether it will be maintained. No code is written until approval is given.
5
Add to this document
Update tokens-lcd.css with the new token, add it to the Token Reference table here, document its CSS class (if a component), and update the version line at the top of this page. The system is only as strong as its documentation.

Button Variants

Size and role extensions of the base .btn family.

btn-md

28px pill. Between .btn-sm (22px) and .btn (full). Shares mono/uppercase DNA. Pair with primary-sm or secondary-sm modifiers.

btn-ghost-sm

Transparent/no-border at rest. Inverts to ink bg on :active and .is-open. Used for "view all" and snooze label triggers.

btn-block

Full-width, 54px tall. Used in modal/sheet footers inside .form-footer.

Legacy icon buttons (deprecated)

Card-context press states

Rule: when an interactive item lives inside a card / sheet / list with rounded corners, its :hover / :active / .is-selected fill must inherit a matching border-radius so the dark fill curves with the parent. Square fills inside rounded containers fight the geometry.

Codified across .sheet-item, .dropdown-option, .up-item. Apply the same rule to any new card-context interactive child.
